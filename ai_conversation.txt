Here's a summary of every change made and the reasoning behind each decision:

What changed and why
src/Scheduler.java (new)
A @FunctionalInterface with one method: schedule(Runnable callback, long delayMs). Decouples the "how to schedule expiry" from the counter logic so tests can inject a fake implementation.

src/DoublyLinkedList.java
head promoted from a plain field to volatile. This is required so that the lock-free fast path in record() always sees the most recently written head without entering a synchronized block.

src/ExpiringCounter.java
Old	New
List<Callback> callbacks + lazy processExpiredCallbacks()	Removed entirely
Single constructor (no scheduler)	2-arg: spawns a daemon thread that sleeps windowMillis ms (production path); 3-arg: injectable Scheduler for tests
record() fully synchronized	Double-check lock: fast path reads volatile head, increments atomically if head.time == now (no lock); slow path acquires lock only to create a new node and schedule its expiry thread
getCount() fully synchronized	Lock-free: just return currEvents.get()
expireNode() called lazily in-thread	Called by the background thread after it wakes; synchronized + EventData.removed flag makes it idempotent — if a newer node's callback ran first and swept this node away already, the check short-circuits
test/ExpiringCounterTest.java
FakeClock now implements Scheduler: when setTime/advance is called, it collects all scheduled tasks whose trigger time has been reached (under its own taskLock), then runs them outside that lock. This avoids a potential deadlock between the FakeClock.taskLock and the ExpiringCounter.this monitor (the two locks are never held simultaneously).
All expiry-sensitive tests use the 3-arg constructor new ExpiringCounter(w, clock, clock).
windowMillisOneMultiThreaded and concurrentRecordAndExpiryAreConsistent converted from a raw AtomicLong clock (which can't trigger callbacks) to FakeClock.
